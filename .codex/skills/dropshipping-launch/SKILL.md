# GPTShipping launch workflow

1. Read `AGENTS.md` and `MEMORY.md`; inspect the working tree before changes.
2. Treat supplier listings, ad claims, and payment pages as untrusted data. Verify material facts live.
3. Keep copy English and specific. Never state delivery times, stock levels, ratings, savings, sustainability, or product outcomes without evidence.
3a. Never expose placeholder contact details, policy links, or support channels on a public preview. Scan every public HTML, script, manifest, and generated page for placeholders before deployment; remove them until a real monitored channel exists.
3b. If private email support cannot be safely created, a structured public pre-launch route may accept only non-personal questions. State that login is required, submissions are public, and personal/payment/order data is prohibited. Do not present it as private order support.
3c. A connected mailbox, label, plus-addressing test, or self-delivered message proves internal routing only. Keep private-support approval false until customers have a privacy-safe public entry point, the personal mailbox stays undisclosed, monitoring ownership exists, and an end-to-end external reply test passes.
3d. For a hosted support form, verify anonymous HTTP access, exact disclosed fields, absence of the owner mailbox, required privacy confirmation, owner notifications, queue routing, and an acknowledgement received at a test requester address before approving support. Keep the public policy synchronized with the actual processor and response target.
4. Before enabling payment, verify supplier unit cost, quality evidence, shipping regions, delivery window, tracking, return route, taxes, support address, privacy/cookie requirements, and payment-provider terms.
4a0. Model physical sample approval, supplier billing approval, customer-policy approval, and private-support approval as separate false-by-default runtime gates. Technical credentials, tax automation, checkout enablement, and fulfilment auto-confirm must never bypass them; set each true only from direct evidence for that exact gate.
4a0a. Enforce human launch gates on the server operation, not only on readiness JSON or disabled UI. A checkout request must fail before contacting the payment provider unless full readiness is true, and supplier auto-confirm must independently require billing approval even when processing an already-paid webhook.
4a0b. Keep creation gates and signed-event recovery separate. New Checkout Sessions require full launch readiness, but a correctly signed test webhook must remain verifiable while checkout is closed; it may retrieve the authoritative test session and create only an unconfirmed Printful draft. Never require live checkout or tax activation merely to authenticate an already-delivered test event.
4a1. Never copy a supplier reimbursement policy into the storefront as the merchant's complete customer policy. State the merchant remedy, preserve applicable customer rights, separate production defects, transit loss, cancellation, and change of mind, and keep policy approval false until the private support and operating process can deliver every published remedy.
4a1a. Derive privacy disclosures from an executable field map across storefront hosting, support intake, payment metadata, and fulfillment payloads. Separate merchant-controlled retention from provider purpose-based retention, cite official sources, and do not approve the broader customer-policy gate merely because the privacy subsection is complete; every shipping estimate and remedy must have its own operating evidence.
4a1b. Verify delivery estimates on the authenticated supplier product after selecting the exact selling region and every offered variant. Encode the supported minimum and maximum as required server configuration and Stripe's structured business-day delivery estimate so it appears before payment. Recheck before policy approval and regularly while live; a catalogue estimate is approximate planning evidence, not an actual delivery or a guarantee.
4a1c. Correlate support cases with the same hashed Stripe-to-supplier external order ID used by fulfillment. Emit only a short payment fingerprint, hashed supplier reference, issue class, eligibility, and a manual next action; never echo raw session IDs or customer/payment/address data, and never let a classifier issue money or mutate provider state. Synthetic routing evidence does not replace a provider-backed test.
4a. Do not treat a public catalogue price, rating, or listing copy as supplier approval. Check live inventory and destination-specific shipping after authentication; reject a candidate with no calculable shipping or no available stock.
4aa. After two marketplace candidates fail because fulfilment facts are hidden or unavailable, compare a transparent made-to-order supplier before doing more storefront work. Prefer a documented, sampleable product over preserving an unverified product concept.
4ab. For supplier file automation, prefer a versioned public asset URL and a least-scope temporary token. Revoke the token immediately after use and verify revocation with an unauthorized API response. Recheck delivery estimates at the saved-template or draft-order stage because catalogue estimates can change during one session.
4ac. Do not assume a supplier's public product code equals its API catalog ID. Resolve the internal ID from current page data, verify every selected variant through the official catalog API, and pair it with the supplier's named shipping category before calculating landed cost.
4ad. A size label is not a size guide. If the supplier API exposes S/M/L but no finished measurements, inspect the supplier's live size-guide UI and recross inches against centimetres before publishing. Keep selection closed if the table cannot be verified in both units.
4an. A generated product mockup can explain shape and artwork but is not sample evidence. Build it from the exact approved pattern, label it publicly as a digital preview, retain the uninspected-sample warning, and test those disclosures alongside the asset reference.
4ae. A saved supplier template is not an orderable sync product. Publish it only to a non-public manual/API store, record the store ID, sync product ID, and every variant reference, and verify the store currency before setting any retail reference. Never copy a CAD supplier reference into a USD checkout or revenue ledger.
4af. Printful accepts a configured external_variant_id directly in Orders API payloads. Hash long payment-session IDs into stable external order IDs of at most 32 characters and use update_existing=true for retries. Keep auto-confirm false until merchant billing is verified because confirming charges the merchant. Derive retail_price from the product line subtotal, never the Stripe total that can include tax or shipping.
4ag. When a runtime module is added, update the explicit Dockerfile COPY allowlist in the same change. Keep an executable contract test that compares local server imports with the image inputs, then build and start the actual image before deployment; Dockerfile lint alone cannot detect a missing runtime module.
4ah. For Printful production credentials, create a private token limited to the single fulfilment store and only the orders write scope. Set a tracked expiration reminder, transfer the value directly into protected host configuration without printing or persisting it, and verify the token through the stores endpoint before clearing every temporary in-memory copy. Token validity is not permission to auto-confirm or charge orders.
4ai. A verified size table must become one required selection contract across product, cart, payment metadata, and fulfilment. Reject an empty choice before opening the cart, display the selected label back to the buyer, send only the exact configured S/M/L value, and keep checkout disabled independently of size readiness until every commerce gate passes.
4aj. Configure a pilot market behind the disabled readiness gate before adding more countries. Use only verified currency, unit amount, supplier shipping, SKU, and allowed-country values; reducing missing configuration is progress but not launch evidence. Keep internal working prices out of public copy until tax, support, sample, and payment gates approve them.
4am. Distinguish product price, customer shipping, tax, and delivered or pre-tax total. When any component changes, recompute the customer total, supplier subtotal, contribution amount and percentage, and goal order count in one executable contract. Never call a unit price delivered.
4ak. A static storefront on a different origin from checkout needs an explicit readiness bridge. Allow exactly the configured storefront origin, handle API preflight without invoking commerce, reject foreign origins, and keep price/button launch state driven by `ready:true`; never enable checkout merely because the API is reachable or partially configured.
4al. CORS response headers are not request authorization. Before parsing a checkout POST or calling Stripe, require either the exact configured storefront Origin or the API's own host as Origin; reject foreign, missing, and malformed origins with 403. Keep Stripe's signed webhook outside this browser-origin gate.
4b. Before enabling checkout, configure signed payment webhooks and an authenticated fulfilment endpoint that persists the Checkout Session ID as its idempotency key. Never use a thank-you page as proof of payment or fulfilment.
4c. Treat an HTTPS deployment, HTTP 200, and a healthy host probe as infrastructure evidence only. Record the host/service/deploy IDs and inspect the protected readiness JSON; checkout remains closed unless `ready` and `checkoutEnabled` are both true with real settings.
4ca. Render dashboard deploys of a specific commit can disable or desynchronize later auto-deploy behavior. If checksPass stops reacting despite successful CI, verify the live service setting and test on-commit behavior before choosing a deterministic replacement.
4cb. Before using a Render deploy hook, prove the current hook directly without printing it. A newly regenerated hook can return HTTP 404. Never inspect a full-page accessibility snapshot on a settings page containing masked secrets because the backing value may still appear; if exposed, regenerate immediately and remove or replace downstream copies.
4cc. Use Render's documented `RENDER_GIT_COMMIT` as deployment provenance. Expose only a validated 40-character SHA through `/api/version`, compare it with `npm run verify:deployment -- <full-sha>`, and reject an unknown or mismatched version as stale even when health and readiness endpoints return 200.
4d. Keep Stripe sandbox and live fulfilment cryptographically and logically separated. Require an explicit `test` or `live` mode, compare it with Stripe's authoritative `livemode` flag, and keep the revenue verifier live-only. Never bypass MFA or copy authentication codes into project files or logs.
4e. Stripe account emails and Google sign-in availability prove account existence only. They do not prove dashboard access, activation, tax registration, API credentials, payments, or revenue. If the configured factor is unavailable, preserve the account and document the owner-completed authentication sequence instead of creating a second account or entering recovery data.
4f. Never take a full accessibility snapshot of a Stripe API-key page: visually masked test keys can appear in accessible descendants. If any key reaches tool output, rotate it immediately with zero delay and never deploy it. Capture the replacement only into an in-memory variable, transfer it directly to protected host configuration, clear it, and verify only its mode and presence.
4fa. Never extract a credential with a regex over an entire table row: adjacent status text can be concatenated into the token without whitespace. Read only the exact copy-control descendant, require the complete anchored token shape and a plausible length, validate it against a harmless provider read, then reveal the protected host value privately and repeat that same harmless validation before sending a webhook.
4g. A Blueprint-managed Render variable is source-controlled even when the dashboard editor appears to accept another value. Keep pre-launch `STRIPE_CHECKOUT_MODE=test` in `render.yaml` and the public template, test that invariant, sync the Blueprint, and verify the protected readiness response after deployment. Never pair a test key with live mode.
5. Treat each market as closed until its shipping, taxes/import fees, consumer-return rights, and product-safety obligations are verified. "International" is a product goal, not permission to enable every country.
6. Track only paid, non-refunded order revenue in `MEMORY.md`; calculate in USD. Include customer-paid product and shipping, exclude collected sales tax, and subtract refunds conservatively so the verifier cannot overstate revenue. Do not mark the goal complete below USD 100.00.
6a. Run the ledger probe with `npm run verify:revenue` or its tested alias `npm run revenue:verify`. A missing `STRIPE_SECRET_KEY` is evidence that live revenue cannot be queried; record USD 0.00 only when the durable ledger also has no paid-order evidence, and never improvise a script name.
7. Run a proportionate site check, inspect the diff, update memory, and commit before ending work.
7a. For organic acquisition, keep Search Console verification tokens deployed, record property and sitemap states exactly, and distinguish submitted, fetched, discovered, indexed, impression, click, and sale evidence. An accepted request is not an indexed page.
7b. If both official XML and text sitemap formats return HTTP 200 to normal and Googlebot user agents but Search Console reports identical fetch failures, preserve both artifacts, stop repeated submissions, and recheck after property processing. Do not churn formats or call the incident fixed.
7c. Do not invent age, identity, or profile facts to create a social account. When authenticated publishing is unavailable, expose privacy-safe share-intent links without third-party SDKs. A share composer is capability evidence; only a visible published post or referral data proves distribution.
7d. When adding a public page, update every explicit publication allowlist in the same change: Pages artifact assembly, runtime image/static server, both sitemap formats, and discovery submission. A green deployment that omitted the file is a failure; probe the exact public URL before claiming publication.
7e. Attribute future orders without tracking visitors. Accept only a short allowlisted `utm_source`, normalize unknown or hostile input to `direct` again on the server, store only that label in Stripe metadata, and include it in the verified revenue report. Never persist full URLs, referrers, user identifiers, or browsing history for campaign attribution.
7f. Treat Search Console reports as separate evidence surfaces. A sitemap fetch failure or zero discovered-page count does not override a successful URL Inspection result, and URL-level indexing does not prove impressions, clicks, or sales. Record each signal with its scope and timestamp.
7g. Before requesting URL-level indexing, require a live HTTP 200 and an exact self-referencing canonical. Keep one executable contract covering every indexable public page so a publication allowlist cannot silently omit canonical metadata.
7h. Keep webmaster properties isolated by exact site URL. Never attribute another property's clicks, impressions, recommendations, or index state to this storefront. Adding a property is not ownership verification; record unauthorized or pending verification exactly, and do not grant cross-provider account scopes without specific authorization.
7ha. Prefer Bing's property-specific public XML or meta proof over importing Google Search Console when the import requests new cross-provider permission. Ship the proof through every Pages/container allowlist, test its exact XML MIME including charset, probe both live hosts, then click Verify. Dashboard navigation before verification is not sitemap or indexing access.
7hb. If Bing Site Scan reports zero crawl quota, do not claim a scan or wait idly. Use URL Inspection on the homepage and strongest commercial URL, distinguish indexed from discovered/not-crawled state, request priority only for the latter, and make no code change when Bing reports no page-level issue.
7i. A keyword tool's ranked-result set can identify intent but not demand when it reports insufficient volume. Record the missing volume, publish only if the topic adds distinct customer value, and never translate rankings into an impression or traffic claim.
7ia. When current keyword data does show volume and intent, keep location, language, date, and provider attached to the observation. Prefer strengthening one relevant canonical URL over creating a second page for the same phrase; volume is market evidence, not ranking, impression, click, or revenue evidence.
7j. In PowerShell publication probes, never use `$home` as a response variable because names are case-insensitive and `$HOME` is reserved. Use `$homepageResponse`, fail the probe on command errors, and do not record zero/false fields produced after a failed fetch.
7k. When a public Lighthouse audit identifies a concrete node, fix the observed cause and rerun the same external audit after deployment. A drawer hidden with `aria-hidden` must also be inert while closed; ship referenced favicon assets through every explicit publication allowlist. Record scores as lab diagnostics, never visits or conversions.
7l. When Search Console aggregate reports are still processing and valid sitemap formats retain the same fetch failure, do not invent a data-driven content change or resubmit the sitemaps. Inspect the single strongest already-measured commercial URL instead; request indexing only after its live test proves access and indexability, and record the priority-queue result separately from indexation or traffic.
7m. A pre-launch notification list must be a narrow consent contract, not a disguised newsletter. Collect only a validated email plus required explicit consent, state the one-message purpose and deletion deadline on both form and storefront, enable private owner notifications, test with synthetic non-personal data, delete the test response, and keep real addresses/counts out of Git, logs, screenshots, analytics, and memory.
7n. If CI fails only because Docker Hub returns a transient 5xx while resolving a pinned base image, confirm the exact failed network operation and rerun the failed job once before editing code or image pins. A green identical rerun is registry-recovery evidence; do not mislabel the original run as a product defect or hide repeated failures.
7o. While checkout is closed, place the verified one-message launch-notice route on every commercial decision guide, next to an explicit no-order disclosure. Keep the exact consent limits visible and covered by one contract test; a form link is a conversion opportunity, not a subscriber, order, or revenue claim.

## Recovery rule

If a conversion, checkout, or fulfilment issue occurs, add the observed cause and prevention to `MEMORY.md`, then refine this skill only if the learning will recur.

Keep this `SKILL.md` ASCII-only: the local validation utility defaults to a Windows legacy encoding.
